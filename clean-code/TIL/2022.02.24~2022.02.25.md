## ✏️ [TIL] Assignment #6

### 🥸 오늘 읽은 범위

4장 주석

### ✅ 책에서 기억하고 싶은 내용

1. 주석은 나쁜 코드를 보완하지 못한다.
2. 코드로 의도를 표현하라!
3. 좋은 주석
    - 법적인 주석
    - 정보를 제공하는 주석
    - 의도를 설명하는 주석
    - 의미를 명료하게 밝히는 주석
    - 결과를 경고하는 주석
    - TODO 주석
    - 중요성을 강조하는 주석
    - 공개 API에서 Javadocs
4. 나쁜 주석
    - 주절거리는 주석
    - 같은 이야기를 중복하는 주석
    - 오해할 여지가 있는 주석
    - 의무적으로 다는 주석
    - 이력을 기록하는 주석
    - 있으나 마나 한 주석
    - 무서운 잡음
    - 함수나 변수로 표현할 수 있다면 주석을 달지 마라
    - 위치를 표시하는 주석
    - 닫는 괄호에 닫는 주석
    - 공로를 돌리거나 저자를 표시하는 주석
    - 주석으로 처리한 코드
    - HTML 주석
    - 전역 정보
    - 너무 많은 정보
    - 모호한 관계
    - 함수 헤더
    - 비공개 코드에서 javadocs

### 🤔 오늘 읽은 소감

- 이번 토픽은 “주석 자체가 나쁘다" 대신 “주석을 최소화 할 수 있는 코드를 고려하자”로 해석했다.
- 하지만 주석을 최소화 하는 것도 과연 옳은 일인가? 라는 생각이 든다.
- 나쁜 주석의 예시처럼 골 때리는 주석(닫는 괄호 주석, 주석 처리한 코드 등)은 물론 피해야 할 것이다.
- 하지만 적절한 순간의 적당한 양의 주석은 오히려 코드로 소통하는 개발자들에겐 생산성을 높여줄 수 있다.
- 나 역시, 코드의 완성도와는 별개로 주석은 굉장히 효과가 있었다. (특히 TODO 주석)
- 또한 모국어가 영어가 아니라면 이름만으로는 어느정도 한계가 있을 것이다.
  - 국내 개발자들에게는 6-7단어 이상의 이름은 오히려 이해하기 어려울 수 있음
  - 네이밍으로 한참을 고민 → 군데군데 다른 이름 → 규칙 늘어남 → 생산성 저하
